# PrivacyTap 课程手册与实验计划设计

## 目标

将 PrivacyTap 当前已经实现和验证的能力整理为一套可直接安装、演示、实验和答辩的课程材料，使教师能够明确判断：

1. 项目发现了一个真实且突出的中间件问题；
2. 方案不是普通日志脱敏，而是请求出站前的全链路隐私治理；
3. 中间件确实位于 AI 客户端和模型服务之间并承担协议适配、隐私转换、状态隔离和可观测性职责；
4. 项目价值可以通过可重复、可量化、有证据等级的实验验证。

## 核心问题

传统日志脱敏只保护“已经产生的日志”，无法阻止原始手机号、身份证号、邮箱、银行卡号、学号或代码凭证被发送给云端模型。简单删除或单向替换虽然降低泄露风险，却会破坏 Codex、Claude Code 的流式文本、文件路径和工具调用参数。

PrivacyTap 的解决方式是在请求离开本机前进行请求级可逆匿名化，只将占位符发送给上游；上游响应回到本机后，再在文本和工具参数中恢复原值。安全归档和 Langfuse 只接收恢复前事件。

## 证据设计

证据按证明力分为四级：

| 等级 | 证据 | 主要证明内容 |
|---|---|---|
| A | 可控上游捕获的实际 HTTP 请求和安全归档扫描 | 原始敏感值确实未离开 PrivacyTap |
| B | 自动化测试、覆盖率、数据集评测和边界切分测试 | 实现正确、可重复且覆盖关键分支 |
| C | 真实 Codex/Claude Code 二进制经过 PrivacyTap 运行 | 真实客户端协议兼容和工具可用性 |
| D | 真实云 API 任务记录 | 外部服务环境下的端到端可用性 |

安全结论至少需要 A+B；“可真实使用”结论至少需要 A+B+C；D 用于增强外部有效性，但不能代替 A，因为云服务通常不提供服务端原始请求证据。

## 验收标准

- 两份核心文档不存在 `TODO`、`TBD` 或伪造实验数据；
- 所有命令与当前 CLI、路径和 Provider 配置一致；
- 明确区分“当前已验证结果”和“需使用个人 API Key 执行的实验”；
- README 可直接访问两份核心文档；
- 旧项目档案和威胁模型不再声称仅支持非流式 Chat Completions；
- 文档契约测试和完整测试套件通过。
